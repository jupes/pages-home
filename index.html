Codebase Guide

1. TL;DR

This repo implements a natural-language AI admin assistant for a mock e-commerce backend. A store admin opens the existing /admin page, types a question or command into an embedded chat panel, and the agent reads orders and products or performs confirmation-gated writes (cancellations, status transitions, price and description updates) against a seeded REST API. Reasoning runs entirely locally via Qwen3 on Ollama — no cloud LLM calls.

The system is four processes wired with concurrently under a single npm run dev:

  1. Mock REST API (starter code, port 3000) — the persistence layer, unmodified except for a tax-calculation bug fix.
  2. MCP server (port 3100) — exposes six tools over Streamable HTTP, owns input validation, the strict order state machine, and REST calls into the mock API.
  3. Agent service (port 3200) — Express app built on Google ADK (@iqai/adk@0.8.5) with a custom Ollama adapter. Owns the /agent/chat HTTP contract, two-step confirmation flow, destructive-tool gating, and think-trace stripping.
  4. Vite dev server (dev-only, port 5173) — serves the React chat island standalone for UI iteration; production builds are static assets served by the mock API.

There are two canonical code paths. A read (e.g. "What is order ORD-1001?") flows from the browser through the agent service, which invokes the LLM, the LLM calls a read tool, the agent forwards that call over MCP to the MCP server, the MCP server fetches from the REST API, and the result unwinds back as chat text. A confirmed write (e.g. "Cancel order ORD-1001") follows the same inbound path but the destructive-tool wrapper intercepts the LLM's tool call, issues a confirmation token, and returns a sentinel. The handler detects the sentinel and responds with a pendingAction payload. The UI renders Approve/Decline buttons. On approval, a second POST carries the token; the handler consumes it and forwards the held action directly to MCP — no LLM involved in the second turn.

The most consequential design decisions: MCP as the tool transport instead of in-process adapters (ADR-7), giving the tool surface independent exercisability; a strict agent-side state machine layered on top of the permissive REST API (ADR-2); scoped two-step confirmation where only irreversible or money-touching actions are gated (ADR-3); and client-sends-history memory with server-side caps (ADR-5), avoiding server-side session persistence.

2. System topology

Mock REST API (port 3000, entry point src/server.ts). The starter Express app serving seeded JSON data from data/seed/. Handles CRUD for orders, products, and promotions across three route files (src/routes/orders.ts, src/routes/products.ts, src/routes/promotions.ts). The API exposes endpoints like GET /api/orders, GET /api/orders/number/:n, GET /api/products, POST /api/orders/:id/status, POST /api/orders/:id/cancel, PUT /api/products/:id, and PUT /api/products/:productId/variants/:variantId. All responses are wrapped in { success: boolean, data: T } envelopes.

Writes are gated behind enableDynamicMode() (in src/utils/storage.ts), which is triggered by the /api middleware when either the HTTP method is PUT/POST/DELETE or the Referer header contains /admin:

src/server.ts:190-198

app.use('/api', (req: Request, res: Response, next: NextFunction) => {
  const referer = req.get('referer') || '';
  if (req.method === 'PUT' || req.method === 'POST' || req.method === 'DELETE' || referer.includes('/admin')) {
    enableDynamicMode();
  }
  next();
});

On first write, seed data is copied from data/seed/ to data/dynamic/ for mutation; data/dynamic/ is gitignored and recreated from seed on next write after deletion. The only change to starter code is a tax-calculation fix in src/routes/orders.ts (three locations: order creation, add-line-item, and update-line-item) where totalPrice was changed to exclude tax so that grandTotal = subtotal - discount + shipping + taxTotal doesn't double-count. The original code computed totalPrice = (unitPrice * quantity) - discount + tax, which caused tax to be counted both in subtotal (sum of totalPrice) and in taxTotal (sum of per-item tax).

MCP Server (port 3100, entry point mcp-server/src/server.ts). Exposes six tools (getOrder, getProduct, cancelOrder, updateOrderStatus, updateProductPrice, updateProductDescription) plus a __ping sanity tool over both Streamable HTTP (/mcp) and legacy SSE (/sse + /messages). Owns all input validation, the strict order state machine, and REST calls via its RestClient. Every failure is normalized into an ErrorEnvelope before leaving the server.

Agent Service (port 3200, entry point agent-service/src/server.ts). Express app with a single endpoint POST /agent/chat. Discovers MCP tools at startup via McpToolRegistry (which wraps ADK's McpToolset). Replaces the four destructive tool names with locally-built confirmation-gated wrappers. Runs one ADK agent loop per standard turn using a shared InMemorySessionService that persists conversation history across requests. Strips <think> traces from assistant output. Manages the ConfirmationStore for two-step writes.

Vite Dev Server (port 5173, dev-only, entry point agent-service/admin-chat-ui/vite.config.ts). Serves the React island standalone with a proxy to the agent service. Production builds emit chat.js and chat.css into public/admin/chat/, which the mock API serves as static assets.

Startup dependency order. The concurrently invocation in package.json starts dev:api, dev:mcp, and dev:agent in parallel, but the agent service's script uses wait-on http://localhost:3000/health http://localhost:3100/health to block until both the REST API and MCP server are healthy before launching. This ordering matters because the agent service's McpToolRegistry discovers tools at first use, and the McpForwarder connects lazily — but the wait-on ensures both dependencies are ready before the agent accepts its first /agent/chat request. Additionally, the agent service itself probes the MCP server's health endpoint on startup (waitForMcpHealth in server.ts, 30 attempts × 500ms) as a defense-in-depth measure, though this can be disabled with AGENT_REQUIRE_MCP_HEALTH=false for testing.

The Vite dev server (port 5173) is started separately via npm run dev:ui and is only needed during UI development. Production builds are pre-compiled via npm run build:ui and served as static assets by the mock API, so the baseline run under npm run dev is three processes, not four.

3. Request flow walkthroughs

Walkthrough A — Read flow: "What is order ORD-1001?"

Step 1. The admin types the question and clicks Send. The useAgentChat hook in the React island fires a fetch POST to http://localhost:3200/agent/chat with the rolling message history and (on subsequent turns) the echoed sessionId.

    setLastError(null);

Step 10. The useAgentChat hook receives the response and processes it through applyResponse: it stores the returned sessionId in sessionIdRef for subsequent turns, sets lastError from the response's error field (if any), sets pending from pendingAction (null for reads), and appends the assistant message to the rolling history. Before storing, the hook applies stripThinkClient — a defensive client-side strip of <think> blocks — to the assistant text. This guards against the .62 regression where unstripped thinking text accumulated in the client's message history, causing the next turn's request body to exceed the server's 8,000-character content cap. The hook then caps the stored history to maxRenderedTurns (default 24) and React re-renders the chat transcript with the new bubble. If assistantThoughts is non-null and non-empty, the bubble includes a collapsed <details> element labeled "Thoughts" so reviewers can inspect the model's reasoning chain.

Step 1–4. Identical to Walkthrough A through agent construction.

Step 5. The LLM decides to call cancelOrder. This name is in DESTRUCTIVE_TOOL_NAMES, so the LLM's tool call is handled by the locally-built wrapper, not the raw MCP tool.

Step 10. The handler enters Path 1 (confirmation second turn). It calls ConfirmationStore.consume(sessionId, token). The store validates: token exists, session matches, TTL not elapsed. On success, it returns the held action and deletes the entry (single-use).

agent-service/src/confirmation/store.ts:48-56

consume(sessionId: string, token: string): ConsumeResult {
  const action = this.entries.get(token);
  if (!action) return { ok: false, code: "INVALID_TOKEN" };
  this.entries.delete(token);
  if (action.sessionId !== sessionId) return { ok: false, code: "MISMATCHED_TOKEN" };
  if (action.expiresAt <= this.now()) return { ok: false, code: "EXPIRED_TOKEN" };
  return { ok: true, action };
}

Step 11. The handler calls deps.forwarder.callTool("cancelOrder", args), which connects to the MCP server over Streamable HTTP and invokes the cancelOrder tool directly — no LLM is involved in this turn.

Step 12. Inside the MCP server, cancelOrder runs the existence preflight (GET the order), then calls canTransition(order.status, "cancelled"). If the order is in a state that allows cancellation (e.g. pending), the tool POSTs to /api/orders/:id/cancel with reason and restockItems. If not (e.g. shipped), it returns a narrated refusal with INVALID_ORDER_STATUS.

4. Component deep-dives

4.1 MCP server — tools barrel

mcp-server/src/tools/index.ts is the single registration site. registerAllTools(mcp, rest) calls each tool's registration function, passing the shared RestClient. Each tool follows the same pattern: a zod schema for input validation, an async handler function that takes (rest, args) and returns a CallToolResult, and a registration function that wires the tool into the McpServer with its name, description, and input schema. Errors are normalized into ErrorEnvelope-shaped text content with isError: true and the envelope as structuredContent so both human-readable and machine-readable representations are available.

getOrder (get-order.ts). Accepts orderNumber (preferred) or orderId. Dispatches a GET to /api/orders/number/:n or /api/orders/:id. Returns the full order document as JSON text content. On 404, returns ORDER_NOT_FOUND.

getProduct (get-product.ts). Accepts sku, productId, or nameQuery with explicit priority: productId > sku > nameQuery. SKU lookup scans the full product listing via GET /api/products?limit=200 (case-insensitive variant match) since SKUs live on variants and are not directly addressable by the REST API. Name lookup performs a case-insensitive substring match on product.name. If exactly one product matches, it's returned; if multiple match, the tool returns an AMBIGUOUS_PRODUCT error listing the candidates so the model can ask the admin which one they meant; if none match, PRODUCT_NOT_FOUND. The productId path is a direct GET to /api/products/:id with no listing scan needed.

cancelOrder (cancel-order.ts). The most complex write tool. First performs an existence preflight via GET to retrieve the order's current state. Then calls canTransition(order.status, "cancelled") from the state machine. If the transition is refused (e.g. order is already shipped), the tool returns the narrated refusal with the ORD placeholder replaced by the actual order number or identifier. If allowed, it POSTs to /api/orders/:id/cancel with reason and restockItems (defaults to true). The response includes previousStatus, newStatus, and whether inventory was restocked. This tool deliberately uses the dedicated /cancel endpoint rather than the generic /status endpoint because the cancel endpoint handles inventory restocking and records the cancellation reason.

updateOrderStatus (update-order-status.ts). Three-stage write: existence preflight via GET, strict canTransition(currentStatus, targetStatus) check, then POST to /api/orders/:id/status with the validated status and optional reason. The targetStatus zod enum deliberately omits cancelled from its allowed values — this forces the LLM to use cancelOrder for cancellations, avoiding two paths for the same intent and ensuring every cancellation goes through the inventory-restocking path. Non-terminal transitions (e.g. pending → confirmed) auto-execute without confirmation; terminal transitions (→ completed, → refunded) require admin approval per ADR-3.

updateProductPrice (update-product-price.ts). Validates price >= 0 before any REST call — this is genuinely additive because the starter REST endpoint accepts negative values without complaint. Resolution follows two paths: if sku is provided, it scans the full product listing (GET /api/products?limit=200) for a case-insensitive variant SKU match, then resolves both the parent productId and the variantId; if (productId, variantId) are provided directly, it GETs the product and finds the variant within it. The write is a PUT to /api/products/:productId/variants/:variantId with the new price. The response reports both previousPrice and newPrice so the model can narrate the change to the admin. Always gated by ADR-3 confirmation because price changes directly touch money.

updateProductDescription (update-product-description.ts). Resolves the product by SKU (scans the listing for a variant with a matching SKU, then uses the parent product's id) or by direct productId. The description is length-bounded at 1–4000 characters via zod validation before any REST call. PUTs to /api/products/:id with the new description field. The response reports previousDescriptionLength and newDescriptionLength so the model can confirm the change. This is the only write tool that auto-executes without confirmation — per ADR-3, text edits are reversible and don't warrant the friction of a confirmation step.

4.2 MCP server — state machine

mcp-server/src/state-machine/can-transition.ts exports canTransition(from: OrderStatus, to: OrderStatus): TransitionResult where TransitionResult is { ok: boolean; reason?: string; errorCode?: string }.

The ALLOWED map defines legal outbound edges per status. Terminal states (cancelled, refunded) have empty outbound arrays. completed is semi-terminal: refund flows remain open.

The function checks four conditions in order: unknown from or to status (INVALID_ORDER_STATUS), same-state no-op (ORDER_ALREADY_IN_STATUS), terminal source (ORDER_TERMINAL), and finally whether the target is in the allowed list.

mcp-server/src/state-machine/can-transition.ts:79-117

export function canTransition(from: OrderStatus, to: OrderStatus): TransitionResult {
  if (!isOrderStatus(from)) {
    return { ok: false, reason: `Unknown current status "${from}".`, errorCode: "INVALID_ORDER_STATUS" };
  }
  if (!isOrderStatus(to)) {
    return { ok: false, reason: `Unknown target status "${to}".`, errorCode: "INVALID_ORDER_STATUS" };
  }
  if (from === to) {
    return { ok: false, reason: `Order is already ${from}; no change to make.`, errorCode: "ORDER_ALREADY_IN_STATUS" };
  }
  if (TERMINAL.has(from)) {
    return { ok: false, reason: `Order is already ${from}, which is final — no further status changes are allowed.`, errorCode: "ORDER_TERMINAL" };
  }
  const allowed = ALLOWED[from];
  if (!allowed.includes(to)) {
    return { ok: false, reason: narrateRefusal(from, to, allowed), errorCode: "INVALID_ORDER_STATUS" };
  }
  return { ok: true };
}

For common admin mistakes (e.g. trying to cancel a shipped order), narrateRefusal returns tailored messages like "ORD is already shipped — please refund instead of cancelling." The ORD token is a placeholder that each calling tool replaces with the actual order number. The legal edges are documented in full in docs/order-state-machine.md (ADR-2). Key rules embodied: no resurrection from terminal states, no skipping fulfillment stages (pending → shipped is refused), no regressions past shipped, on_hold can resume to any active pre-fulfillment state, and completed is semi-terminal (refund flows remain open).

4.3 MCP server — REST client

mcp-server/src/rest-client.ts is the sole HTTP client for the system. Three concerns:

The synthetic Referer: http://localhost:3000/admin header is set on every request (configured via McpConfig.apiReferer). This satisfies the starter's enableDynamicMode() middleware, which gates writes on either a write HTTP method or a referer containing /admin.

mcp-server/src/rest-client.ts:44-49

const headers: Record<string, string> = {
  "content-type": "application/json",
  referer: this.cfg.apiReferer,
  ...req.headers,
};

Retry policy: idempotent methods (GET, HEAD, OPTIONS) retry once on 5xx with linear backoff. POST/PUT/DELETE never retry — the two-step confirmation gives the admin explicit re-issue control. Timeout: 10 seconds via AbortController.

Response normalization: the mock API wraps responses as { success, data, meta }. The client unwraps data when present. Failures are normalized into ErrorEnvelope with remediation hints keyed by HTTP status (404 → "Double-check the order number or SKU"). Network exceptions (connection refused) become UPSTREAM_UNREACHABLE; AbortError from timeout becomes UPSTREAM_TIMEOUT. This ensures the agent layer never has to interpret a raw HTTP error — every failure is a structured envelope with a humanMessage the model can paraphrase back to the admin and an errorCode the UI can render in the footer.

mcp-server/src/rest-client.ts:126-137

private envelopeFromException(err: unknown): ErrorEnvelope {
  const isAbort = (err as { name?: string })?.name === "AbortError";
  return {
    humanMessage: isAbort
      ? "The store API didn't respond in time."
      : "Couldn't reach the store API.",
    remediationHint: "Try again, or check that the API process is running.",
    errorCode: isAbort ? "UPSTREAM_TIMEOUT" : "UPSTREAM_UNREACHABLE",
    details: { error: String(err) },
  };
}

4.4 MCP server — transport

mcp-server/src/server.ts creates the Express app and wires the MCP server via attachDualTransport from mcp-server/src/transports.ts. This function mounts two transport endpoints:

Streamable HTTP on /mcp — the modern transport used by ADK, MCP Inspector, and Claude Desktop. Each client session gets its own StreamableHTTPServerTransport keyed by the mcp-session-id header.

Legacy SSE on /sse (downstream) + /messages (upstream) — for interop with older MCP clients.

Both transports share the same McpServer factory, so registered tools appear identically regardless of which transport a client uses. The tools/list discovery handshake is handled by the MCP SDK: the client sends initialize → notifications/initialized → tools/list, and the server responds with the registered tool schemas (including zod-derived JSON Schema for each tool's input).

The choice of Streamable HTTP over stdio is deliberate (documented in ADR-7): stdio is designed for subprocess-managed servers, but these processes are already managed by concurrently. HTTP gives cleaner process independence, better observability, and compatibility with MCP Inspector and third-party MCP clients. The legacy SSE transport is retained for interop but is not used by the agent service in production.

mcp-server/src/transports.ts:30-51

  const sessionId = (req.headers["mcp-session-id"] as string) || undefined;
  let transport: StreamableHTTPServerTransport;
  if (sessionId && streamableTransports[sessionId]) {
    transport = streamableTransports[sessionId];
  } else {
    transport = new StreamableHTTPServerTransport({
      sessionIdGenerator: () => randomUUID(),
      onsessioninitialized: (id) => {
        streamableTransports[id] = transport;
      },
    });
    transport.onclose = () => {
      if (transport.sessionId) delete streamableTransports[transport.sessionId];
    };
    const server = buildServer();
    await server.connect(transport);
  }
  await transport.handleRequest(req, res, req.body);
});

4.5 Agent service — runtime architecture

The .59 refactor plan (docs/plans/adk-session-refactor.md) proposed an AgentRuntime class as a persistent wrapper around InMemorySessionService, LlmAgent, and Runner. That plan was deferred — the class file agent-service/src/llm/agent-runtime.ts does not exist.

Instead, the handler in agent-service/src/chat/handler.ts implements the refactor's core insight directly: a single shared InMemorySessionService (instantiated once per process in server.ts) persists across requests. The handler looks up or creates a session for each request's sessionId, then constructs a fresh LlmAgent and Runner per request — but crucially, the same SessionService is reused, so ADK threads prior turns into LlmRequest.contents as proper chat-completion messages.

The single-user model uses appName: "admin_agent" and userId: "admin" as constants. These, combined with the client's sessionId, form the primary key into the session service.

4.6 Agent service — chat handler

agent-service/src/chat/handler.ts (~335 lines) is the largest file in the service. The high-level control flow:

  1. Validate the request body against ChatRequestSchema (zod schemas in chat/contract.ts). On failure → 400 with VALIDATION_ERROR.
  2. Mint or echo a sessionId. If absent, newSessionId() generates sess_<uuid>.
  3. If body.confirm is present → Path 1 (confirmation second turn):
    • If decision === "decline" → consume (to remove) the token, return DECLINED envelope.
    • Otherwise → ConfirmationStore.consume(sessionId, token). On INVALID_TOKEN, EXPIRED_TOKEN, or MISMATCHED_TOKEN → 409 with the appropriate envelope.
    • On success → call forwarder.callTool(action.tool, action.args) directly. Extract text content from the MCP result. Return 200.
    • If no forwarder (test path) → acknowledge without dispatching.
  4. If body.confirm is absent → Path 2 (standard turn):
    • trimToWorkingWindow caps the message array.
    • Validate that the last message is from the user.
    • Look up or create the persistent ADK session.
    • Discover tools from ToolRegistry.list(). Filter out destructive names; replace with gated wrappers from buildDestructiveToolHandles.
    • Build LlmAgent + Runner. Run runner.runAsync with the new user text.
    • The event loop concatenates only text parts (tool-call/response events have no .text, yielding empty strings).
    • Check for pending actions: first from the issued recorder (authoritative), then from extractPendingAction (text fallback).
    • splitThink separates thoughts from visible text.
    • If pending → strip marker from visible text, build pendingAction block, return 200.
    • Otherwise → return 200 with clean assistantText and optional assistantThoughts.

The system instruction text (defined inline in the handler, lines 219-234) includes explicit guidance against Hermes-style JSON tool calls — a Qwen3 regression observed in .63 where the model printed {"name":"cancelOrder","arguments":{...}} as assistant text instead of using the function-calling channel. It also includes a cancellation-defaulting instruction that tells the model to dispatch cancelOrder immediately with reason: "Customer requested" when the admin doesn't supply a reason, preventing the model from looping (trying to look the order up first or asking for a reason).

The handler also manages agent-originated error copy via agent-service/src/errors/copy.ts, which exports factory functions for six error envelopes: VALIDATION_ERROR, CONFIRMATION_INVALID, CONFIRMATION_EXPIRED, CONFIRMATION_MISMATCHED, CONFIRMATION_DECLINED, and INTERNAL_ERROR. Each produces a consistent ErrorEnvelope with admin-friendly humanMessage and remediationHint text. These are distinct from the MCP-originated envelopes (which flow through from the RestClient unchanged).

4.7 Agent service — destructive-tool wrappers

agent-service/src/tools/destructive-tools.ts defines DESTRUCTIVE_TOOL_NAMES as a ReadonlySet of exactly four entries: updateOrderStatus, cancelOrder, updateProductPrice, updateProductDescription.

executeDestructive(name, args, deps) is the core function. If requiresConfirmation(name, args) returns true, it issues a token via ConfirmationStore.issue, builds a PendingActionResult, fires deps.onPending, and returns the PENDING_MARKER sentinel ("__PENDING_CONFIRMATION__") concatenated with the JSON payload. If false, it forwards to the live MCP server via McpForwarder.callTool and returns the tool's text result.

The gated vs. auto-execute split (from confirmation/predicate.ts): cancelOrder and updateProductPrice always gate. updateOrderStatus gates only when targeting terminal statuses (cancelled, completed, refunded); non-terminal moves auto-execute. updateProductDescription always auto-executes (text edits are reversible).

extractPendingAction(text) is a fallback parser that finds the PENDING_MARKER in free-form prose and extracts the JSON payload by depth-counting braces. The LLM tends to summarize tool results into its prose rather than echoing them verbatim, so the sentinel may appear embedded mid-sentence rather than at the start. The depth-counting approach handles nested JSON objects without requiring a fragile regex. Returns null if the marker is absent, no opening brace follows, braces are unbalanced, or the parsed JSON doesn't contain the expected marker field. In practice, the onPending recorder callback is the authoritative signal (it fires from the wrapper itself, regardless of what the LLM does with the tool result text), and extractPendingAction serves as a belt-and-suspenders fallback.

4.8 Agent service — confirmation store

agent-service/src/confirmation/store.ts implements ConfirmationStore. issue(sessionId, tool, args) generates a tok_<uuid> token, stores the action with createdAt and expiresAt (TTL defaults to 5 minutes), and returns { token, expiresAt }.

consume(sessionId, token) validates in order: existence (INVALID_TOKEN), then deletes the entry unconditionally (even on failure, to prevent replay — this is the "delete first" pattern that ensures a mismatched or expired token cannot be retried), checks session match (MISMATCHED_TOKEN), and checks TTL (EXPIRED_TOKEN). On success returns the held PendingAction containing the tool name, arguments, session affinity, and timestamps. The store also exposes a size() method used by integration tests to assert that tokens are properly cleaned up after consumption, decline, or expiry. The default TTL is 5 minutes (configurable via CONFIRMATION_TTL_MS), and the clock is injectable via the constructor for deterministic testing of expiry behavior.

The predicate lives in agent-service/src/confirmation/predicate.ts. requiresConfirmation(toolName, args) is a pure function that encodes ADR-3's scope.

agent-service/src/confirmation/predicate.ts:23-37

export function requiresConfirmation(
  toolName: string,
  args: Record<string, unknown>,
): boolean {
  if (toolName === "cancelOrder") return true;
  if (toolName === "updateProductPrice") return true;

  if (toolName === "updateOrderStatus") {
    const target = (args?.targetStatus ?? args?.status) as string | undefined;
    if (!target) return true; // be safe if shape is wrong
    return TERMINAL_ORDER_STATUSES.has(target);
  }

  return false;
}

TERMINAL_ORDER_STATUSES is the set { "cancelled", "completed", "refunded" }. If the args don't specify a targetStatus at all, the predicate defaults to requiring confirmation (defensive). ADR-3 scopes confirmation to "anything the admin can't cleanly undo or that directly touches money."

4.9 Agent service — MCP forwarder and registry

McpToolRegistry (agent-service/src/tools/mcp-registry.ts) wraps ADK's McpToolset. On list(), it lazily initializes the toolset with transport: { mode: "sse", serverUrl } (the bundled SDK client negotiates Streamable HTTP first). The returned BaseTool instances are what the LLM agent sees for read tools. The ToolRegistry interface (agent-service/src/tools/registry.ts) abstracts this behind list() and close(), enabling InProcessToolRegistry for tests.

McpForwarder (agent-service/src/tools/mcp-forwarder.ts) is a separate thin MCP client used for two purposes: auto-executing destructive tools that pass the confirmation predicate, and executing held actions on the approval second turn. It exists separately from McpToolset because those call sites already know exactly what to invoke — they don't need ADK's agent loop. The forwarder lazy-connects via StreamableHTTPClientTransport and caches the Client instance. Connection is guarded against concurrent initialization by storing the in-flight Promise.

agent-service/src/tools/mcp-forwarder.ts:43-48

async callTool(name: string, args: Record<string, unknown>): Promise<CallToolResult> {
  await this.connect();
  if (!this.client) throw new Error("MCP forwarder failed to initialize");
  const result = (await this.client.callTool({ name, arguments: args })) as CallToolResult;
  return result;
}

The Ollama LLM factory (agent-service/src/llm/ollama.ts) wraps ollama-ai-provider-v2 (pinned at v1.5.5 for LanguageModel V2 compatibility with ADK 0.8.5) and returns an AiSdkLlm. The pin avoids the LanguageModel V2/V3 spec mismatch that v3.5.0 would re-introduce. A side effect of this pin is that Qwen3's message.thinking field (containing chain-of-thought) is dropped by the provider before the AI SDK sees it, which is why assistantThoughts is often null in practice — the splitThink extractor handles the inline <think> markers when the model does emit them. Investigation is documented in docs/thoughts-dropdown-investigation.md.

4.10 Agent service — think-trace handling

agent-service/src/chat/strip-think.ts handles four Qwen3 output shapes in a compositional pipeline.

splitThink(input) returns { visible, thoughts }. It processes three passes sequentially, each a no-op when its target is absent:

  1. Paired tags: regex /<think>([\s\S]*?)<\/think>/gi captures thought content and removes the blocks.
  2. Orphan open: regex /<think>([\s\S]*)$/i catches an unterminated <think> block at the end of the string.
  3. Bare close: if any </think> tags survive passes 1 and 2, the text before the last </think> is treated as thought content, and only the text after it becomes visible. This handles the case where Qwen3 starts thinking implicitly (no opening tag) and only marks the end.

The .62 regression was a mixed pattern: <think>cycle1</think>cycle2</think>final. Pass 1 consumed the paired block but left cycle2</think>final. Without pass 3, the leftover rode along as visible text, and on the next turn the 8000-char content cap rejected it. The compositional design fixed this: pass 3 now catches any bare </think> that pass 1 didn't consume.

stripThink(input) is a backward-compatible wrapper that returns just visible.

agent-service/src/chat/strip-think.ts:47-76

export function splitThink(input: string): { visible: string; thoughts: string } {
  if (!input) return { visible: "", thoughts: "" };
  const captured: string[] = [];
  let body = input;
  body = body.replace(THINK_PAIR, (_match, inner: string) => {
    captured.push(inner.trim());
    return "";
  });
  if (body.includes(OPEN_TAG)) {
    body = body.replace(ORPHAN_OPEN, (_match, inner: string) => {
      captured.push(inner.trim());
      return "";
    });
  }
  if (body.includes(CLOSE_TAG)) {
    const lastClose = body.lastIndexOf(CLOSE_TAG);
    const thoughtSection = body.slice(0, lastClose);
    body = body.slice(lastClose + CLOSE_TAG.length);
    for (const piece of thoughtSection.split(CLOSE_TAG)) {
      const trimmed = piece.trim();
      if (trimmed.length > 0) captured.push(trimmed);
    }
  }
  return finalize(body, captured);
}

5. Design decisions (ADR summary)

ADR Title Decision Rationale
ADR-1 Same repo, separate processes, concurrently Four processes in one repo, started via concurrently with health-gated startup. The assignment draws three layers and names separation of concerns as an evaluation axis; colocating would blur that boundary.
ADR-2 Strict agent-side transition subset The MCP server enforces a documented state machine stricter than the permissive REST API, refusing transitions before any write. The assessment explicitly requires transition validation; mirroring a permissive backend 1:1 means no real validation story.
ADR-3 Scoped two-step confirmation Destructive/irreversible writes and price changes require admin approval; description edits and non-terminal status moves auto-execute. Confirming every text edit adds friction without safety payoff; anything that can't be cleanly undone or touches money is gated.
ADR-4 JSON transport, streaming deferred Each agent turn is a single HTTP round-trip; no SSE streaming in v1. With two-step confirmation, turns are short; streaming's payoff on long generations doesn't apply.
ADR-5 Client-sends-full-history Client sends message history each turn; server caps at 16 messages and trims to 8 turn pairs. Zero server-side conversation state; simplicity and no persistence requirement. (Post-.59, the InMemorySessionService is the actual source of truth, but the body cap remains as validation hygiene.)
ADR-6 Qwen3 smallest-viable first Start at qwen3:4b, scale up only if tool-calling reliability requires it; qwen3:8b is the current default. Assignment requires Qwen3 ≤30B; tool-calling quality drops on tiny models but large models won't run on 8 GB.
ADR-7 Tool surface ships as MCP server The agent's tools are served by a local MCP server over Streamable HTTP, not as in-process adapters. Mirrors real-world agent tooling (Claude Desktop, Claude Code); tool surface is independently exercisable by any MCP client; clean protocol boundary.

7. Testing strategy overview

See docs/test-coverage-review.md for the full test inventory (264 tests across 24 files). This section describes the strategy layers.

Pure-function unit tests cover the state machine (can-transition.ts — exhaustive transition matrix, 15 tests) and think-trace stripping (strip-think.ts — all four Qwen3 output shapes including the .62 mixed-pattern regression, 20 tests). These are the cheapest tests: no I/O, no mocks, deterministic. They earn their keep by pinning exact behavioral contracts on the two components most likely to regress under model or framework changes.

Fake-fetch isolation tests cover the six MCP tools (get-order, get-product, cancel-order, update-order-status, update-product-price, update-product-description — 28 tests total) and the RestClient (6 tests). Each test injects a fake fetch into RestClient, so the validation layer, state-machine integration, and HTTP wiring are exercised without a live API. This layer proves that each tool's input validation, error mapping, and REST call shape are correct independently.

In-process REST+MCP integration tests (9 tests in confirmation-flow.test.ts) boot the starter REST API and MCP server in-process and drive the two-step confirmation flow end-to-end: token issuance, approval, decline, expiry, auto-execute, state-machine refusal, MCP-Inspector parity. No LLM is involved — a deterministic driver calls executeDestructive and the forwarder directly. This layer proves the plumbing works.

Stub-LLM contract tests cover the /agent/chat HTTP contract (10 tests) and multi-turn history threading (3 tests). The contract tests use a stub LLM that's never actually invoked — the handler's structural behavior (validation, session minting, confirmation dispatch) is under test. The multi-turn tests use a recording stub BaseLlm that captures LlmRequest.contents across three sequential POSTs, proving the shared InMemorySessionService threads history correctly.

Live-Ollama end-to-end tests (2 tests, auto-skipped when Ollama is unreachable) drive the full stack with a real Qwen3 model. The read test (end-to-end-read.test.ts) boots the starter REST API, MCP server, and agent service in-process, then POSTs a prompt like "Use the getOrder tool to look up ORD-1001" and asserts the response contains the order number. The cancel test (end-to-end-cancel.test.ts) drives the full two-step confirmation flow: turn 1 prompts cancellation, turn 2 sends the approval with the held token, and the test verifies the order's REST status actually becomes cancelled. The cancel test is intentionally lenient — it accepts a soft fallback if the model doesn't emit pendingAction (passing as long as the assistant text mentions the order number) — because small-model tool-calling is variable and a strict assertion would make the test flaky on different hardware.

8. Known gaps and open items

These are gaps acknowledged in the existing documentation (docs/test-coverage-review.md and docs/plans/adk-session-refactor.md).

No per-session event cap. The InMemorySessionService accumulates events indefinitely on a single sessionId. A long-running chat will eventually exceed the model's context window. The refactor plan sketches a 32-event cap with tool-call/response pair preservation at the boundary; it was scoped out of .59 to keep that commit small. Impact: only matters for extended single-session chats; restarting the process or rotating sessionId is a workaround.

No live-LLM regression test for multi-turn intent threading. The .59 refactor is validated by structural assertions (stub LLM proves history reaches LlmRequest.contents), but no automated test proves qwen3:8b actually reasons over that history correctly. Verification depends on manual UI testing. Impact: a model or provider regression could silently break follow-up turns.

No automated test of the UI's defensive stripThinkClient. The bundle integrity tests confirm the file ships and references the right mount ID. The strip function itself is not exercised by any test. Impact: a regression in the client-side strip could re-introduce the .62 content-cap overflow.

tools/list discovery test covers only Phase 3 tools. The four Phase 4 destructive tools are registered but not asserted in the MCP discovery test. Impact: a registration regression would not be caught at the discovery level (though individual tool tests would still fail).

buildContextualPrompt is deprecated but not deleted. Scheduled for removal in .60 after live UI verification of .59. Five tests cover it in session.test.ts. No production callers exist. Impact: dead code; harmless but clutters the test count.

The agent service config covers the Express port (AGENT_PORT, default 3200), UI origin for CORS (UI_ORIGIN, default http://localhost:3000), MCP server URL (MCP_URL, default http://localhost:3100/mcp), Ollama base URL and model (OLLAMA_BASE_URL, OLLAMA_MODEL defaulting to qwen3:8b), and confirmation token TTL (CONFIRMATION_TTL_MS, default 5 minutes). The MCP server config covers its own port (MCP_PORT, default 3100), the REST API base URL (API_BASE_URL, default http://localhost:3000), the synthetic referer header (API_REFERER, default http://localhost:3000/admin), request timeout (API_TIMEOUT_MS, default 10 seconds), and retry policy (API_RETRY_MAX defaulting to 2 attempts for idempotent requests, API_RETRY_BASE_MS defaulting to 250ms linear backoff). Both configs accept an env override object for test isolation — no .env file parsing, no dotenv dependency, just plain process.env reads with defaults.
